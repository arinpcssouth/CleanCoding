Multi Layers - Tier Application
A little bit about architectural thinking

Here I will try to focus on the logical layers rather than physical tiers. What you see in this demo is not a structure I would recommend for your small application. Everything comes with a cost and this is a very important factor you would need to consider when trying to scaffold your base product architecture.
As a software architect, it is very important to consider the Cost / Quality / Time triangle before writing your very first sketch diagram on the paper.

The following is a quick diagram showing how different layers of the application interact with each other. In another word, my service layer has no idea about the concrete implementation of the Business and Repository layers. The concrete implementation can be easily switched without bothering the parent layers.

Unit Testing
Unit Tests

My recommendation is to always use unit tests for your business logic layer. However, it is important to use mock frameworks to make sure your other layers are not involved in the unit testing. Otherwise, your tests will be slow and you will start ignoring and skipping the tests overtime and it will result a poor outcome for your project.

Service Layer Tests

Use them as needed. For small projects, you may not need to test every single service API. My recommendation would be to do a test on critical APIs that have significant impact on your application.
Remember the triangle and act accordingly :)

Integration Tests

For enterprise applications, these unit tests can be set to run as a night job. These tests are the most time consuming since it hits your data layer.
You would need to take extra precaution when performing this test. A recommendation would be to run the integration tests against a test database to avoid data loss and corruption.

Token Validation on WebAPI

Most service oriented applications function with tokens getting passed to the services. Wouldn’t it be nice to ask the client to include the token in the message header and not to pollute the main request model (parameter) with an unrelated concept?

There is a solution for that. You can create your own Action Filter Attribute in Web API and validate the token in the request header and reject the call before hitting the action method. 

                        
        [HttpGet]
        [ValidateToken]
        public HttpResponseMessage GetClassrooms()
        {
            var res = classroomFactory.GenerateClassrooms();
          
            return Request.CreateResponse(HttpStatusCode.OK, res);
        }
                    
                        
namespace ServiceLayer.Controllers.ActionFilters
{
    //see NinjectWebCommon.cs for more details. There is a reason this empty class i shere. 
    public class ValidateTokenAttribute : ActionFilterAttribute
    {
    }
    public class ValidateTokenFilter : AbstractActionFilter
    {
        public override bool AllowMultiple { get { return true; } }
        
        
        ISession session { get; set; }
        public ValidateTokenFilter(ISession session)
        {
            this.session = session;
        }
        public override void OnActionExecuting(HttpActionContext actionContext)
        {
            string headerToken = "";
            try
            {
                headerToken = actionContext.Request.Headers.GetValues("Token").FirstOrDefault();
            }
            catch (Exception)
            {
                throw new HttpResponseException(HttpStatusCode.Unauthorized);
            }
            if(session.GetSessionByToken(headerToken).Token == new Guid())
                throw new HttpResponseException(HttpStatusCode.Unauthorized);
        }
    }
}

But Wait!!! How do I take care of Dependency Injection there???

We have a big problem here. To validate the token in action filter, you would need the concrete Business Layer to be injected to the filter.

I encountered this problem three years ago. It was not easy but I finally found a solution to resolve the problem. Most solutions on the internet even today do not solve the problem and is very confusing. Here is how it goes…

You need to create an action filter that inherits from AbstractActionFilter instead of ActionFilterAttribute. Then you create an empty attribute for your token validation.

The magic happens in NinjectWebCommon.cs

By adding this line in RegisterServices method. 

    kernel.BindHttpFilter<ValidateTokenFilter>(FilterScope.Action).WhenActionMethodHas<ValidateTokenAttribute>();
                    
                     
 private static void RegisterServices(IKernel kernel)
        {
            //Here you can easily switch between repository implementaitons. Thanks to Dependency inversion principle. 
            //kernel.Bind<IAuthenticationRepository>().To<RepositoryConcrete.AuthenticationRepository>();
            kernel.Bind<IAuthenticationRepository>().To<RepositoryConcrete.InMemory.AuthenticationInMemoryRepository>();
            kernel.Bind<ISession>().To<BusinessConcrete.Session>();
            kernel.Bind<IClassroomFactory>().To<BusinessConcrete.Factories.ClassroomFactory>();
            //this is to enable DI in action filters. Basically we bind the filter to an empty attribute class. 
            kernel.BindHttpFilter<ValidateTokenFilter>(FilterScope.Action).WhenActionMethodHas<ValidateTokenAttribute>();
        }
Generics
< T > - Covariance and Contravariance

My intend here is not to explain what Generics are but I want to introduce a very specific feature in generics that might come handy in some complex applications. This specific situation is when you are trying to use polymorphic behaviors in generics.

Let's start with an example. In our example, Teacher is derived from Staff and Staff is derived from Person.

In object oriented world, it is perfectly okay to pass a more derived object to a method that accepts a less derived object. For example, a method that accepts a Person as a parameter. It would be perfectly okay to pass a Teacher object to that method. Since teacher is a person. However, in Generic world, the compiler seems to have some problems when trying to pass a less derived generic type parameter to the generic class. This is because the compiler wants to know the direction the generic type parameter is going to be used. This is done by adding “in” or “out” modifiers to the definition.

When “out” modifier is used (Covariant), the methods inside the interface are allowed to return a type that is more derived specified by the generic type parameter. Basically, the generic type parameter is used as an output here.

On the other hand, Contravariance “in” works in the opposite direction. When using “in” modifier you are not allowed to return a more derived type than the generic type parameter and you can only consume the more derived type inside the class but you cannot return it. Which is being used as an input parameter here.

Note that both “out” and “in” modifiers are only allowed in delegates or interfaces.

In the code below you will notice if we remove the “in” or “out” keywords from our interfaces, the test will fail. 

    namespace RepositoryAbstract
    {
        //in
        public interface IRegistrationWriteRepository<in T>

        public interface IRegistrationReadRepository<out T>

            IEnumerable<T> GetEveryone();

        public interface IRegistrationRepository<T> :IRegistrationReadRepository<T>, IRegistrationWriteRepository<T>

        }

UI / UX
The Face of Your Product
Why matters?

User interface and experience considerations are as important as your back-end architecture. You can have the world’s best software product but without proper presentation, your product will simply be unusable.

The winner of UX vs. UI is the UX.

Before going fancy, let's give the user an easy interface to operate. Minimum clicks, do things where expected and ask real users for feedback.
Don't become defensive on the feedback you get. Act on it and put yourself in users’ shoes!

Architect your user interface. Leverage MVVM, MVP and similar patterns

Leverage design patterns in your front-end applications. Nowadays there are thousands of ways you can present your application to the end user and is not limited to web and mobile technologies. TVs, Fridges, Cars, Watches and more…

Managing and maintaining all the UI interfaces can soon become a nightmare if not handled properly.
